## 2026-04-24 — EarlyValidation::ResourceExistenceCheck エラー修正

### 問題
`deploy_mfp.sh` でデプロイすると changeset 作成時に以下のエラーが発生：
`The following hook(s)/validation failed: [AWS::EarlyValidation::ResourceExistenceCheck]`

### 根本原因
SAM `Events` の `ScheduleV2` で `GroupName: !Ref SyncScheduleGroup` を使うと、SAM transformer が `DependsOn` から `SyncScheduleGroup` を剥落させ、かつ EarlyValidation が GroupName の実在チェック時にグループ未作成と判断して失敗していた。

### 変更ファイル
- `template-mfp.yaml`
  - SAM `Events` からスケジュール定義を完全に削除
  - `HourlySchedule` / `DailySchedule` を `AWS::Scheduler::Schedule` 独立リソースとして再定義
    - `DependsOn: [SyncScheduleGroup, SyncFunction]` を直接付与
    - `GroupName: !Ref SyncScheduleGroup` を使用
    - `Target.Arn` / `Target.RoleArn` を明示指定
  - `SchedulerInvokeRole`（IAM ロール）を追加: EventBridge Scheduler → Lambda 呼び出し権限
  - DailySchedule の cron を `cron(59 14 ...)` → `cron(59 23 ...)` に修正（23:59 JST）

## 2026-04-26 — MFP セッショントークン無効化対策（Tier 1）

### 背景
MFP セッションクッキー (`__Secure-next-auth.session-token`) が毎日〜数日に1回無効化される問題。
2時間間隔に伸ばしても解消せず、頻度のみが原因ではないと判断。

### 方針
TLS 指紋偽装等のグレー対策は採用せず、「実ブラウザに近づける」Tier 1 対策のみ実施。
計画ファイル: `/home/node/.claude/plans/myfitnesspal-fluffy-conway.md`

### 変更内容

#### `src/asken_myfitnesspal_sync/myfitnesspal_client.py`
- ブラウザヘッダー全面拡充 (Chrome 131 UA + Sec-Fetch-* + Sec-Ch-Ua-* + Accept-Language 等)
- 認証フローを「Cookie → diary GET (warm-up) → auth_token GET」に変更
  - `?refresh=true` クエリ撤廃（毎回リフレッシュは bot 検出シグナル）
  - diary HTML を `_diary_cache: dict[date, ...]` にキャッシュ、書き込み後 try/finally で破棄
- POST に Origin / Referer / Sec-Fetch-* を付与（`add_meal_entry`, `delete_meal_entries`）
- 失効検知の集中化: `_mfp_request_with_retry` で 401/403 → `MfpAuthError` を一元処理
- 200 OK でも login redirect / Cloudflare チャレンジ HTML を検出する `_raise_if_unauthenticated_html`
- `_api_headers(target_date)` / `_form_post_headers(target_date)` 引数化（init 時の date と書き込み時の date 不整合を解消）
- `MyFitnessPalClient(cookie, target_date)` シグネチャに変更

#### `src/asken_myfitnesspal_sync/sync.py`
- あすけんが当日の食事データを 0 件返した場合は MFP 認証もスキップ（`MyFitnessPalClient` を構築しない）

#### `src/asken_myfitnesspal_sync/handler.py`
- `MfpAuthError` 捕捉 → `_notify_mfp_auth_failure` で SNS publish → 再 raise
- boto3 SNS クライアントの遅延シングルトン
- SNS publish 失敗は logger.exception でログ出力して swallow（二重失敗回避）

#### `template-mfp.yaml`
- スケジュール: HourlySchedule (24/日) → MealTimesSchedule `cron(30 8,13,19)` JST (3/日) + DailySchedule (1/日)
- 全スケジュールに `FlexibleTimeWindow: FLEXIBLE, MaximumWindowInMinutes: 10` を付与
- `MaximumEventAgeInSeconds: 60 → 900`（jitter window と整合）
- `MfpAuthAlertTopic` (SNS, KMS 暗号化) + Lambda IAM `sns:Publish` 権限 + 環境変数 `MFP_AUTH_ALERT_SNS_TOPIC_ARN`

#### `pyproject.toml`
- dev 依存に `boto3-stubs[secretsmanager,sns]` を追加

#### テスト
- `test_myfitnesspal_client.py`: 既存 `_API_URL` インポートエラーを修正、`_make_client(diary_html)` ヘルパー導入
  - 新ヘッダー、新フロー、403/Cloudflare 検知、cache 動作、target_date 引数化を全網羅
- `test_sync.py`: 欠食スキップ・MFP 構築引数の検証テスト追加
- `test_handler.py`: SNS 通知経路テスト 4 件追加 (`TestMfpAuthFailureNotification`)

### 検証結果
- pytest: 326 全パス、警告 0 件
- ruff: All checks passed
- mypy: Success

### evaluator 評価サイクル
- 計 4 回の評価サイクルを実施。Critical 2 件・Major 10 件・Minor 多数を順次解消。
- 4 回目で新たに指摘された Major 2 件（下記）はユーザー判断によりスコープ外・別タスク扱いとして本タスクを完了。

### 別タスクとして残した Major 指摘
- **Major A**: 丸め不一致による無限上書きループの可能性
  - `add_meal_entry` の `round(calories)` / `round(_, 1)` と `_is_same_nutrition` の `is_nutritionally_equal(rel_tol=1e-5)` の不一致
  - あすけん側に小数 2 桁以上の値が含まれる場合に毎回 delete→re-add がループする可能性
  - 既存挙動であり今回の対策スコープ外。別タスクで `_is_same_nutrition` 比較側にも丸めポリシーを適用すること
- **Major B**: 403 一律 MfpAuthError 化による過剰アラート
  - CSRF 不一致 / WAF ソフトブロック等の一時的 403 でも即 SNS 通知してしまう
  - ユーザー要求「失効を即時通知」と「誤検知耐性」のトレードオフ
  - 別タスクで「N 回連続 403 のみ MfpAuthError 化」「食事区分単位の WARNING に格下げ」等を検討

### 重要事項
- `_mfp_request_with_retry` の 401/403 集中処理により全 HTTP 経路でクッキー失効が検知される
- diary HTML キャッシュは書き込み後（`add_meal_entry` / `delete_meal_entries`）に try/finally で破棄してドリフト回避
- 本番運用後 1〜2 週間で失効頻度を観察し、改善されない場合は Tier 2（curl_cffi での TLS 指紋擬態）を再検討
